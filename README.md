### Фича: Публикация товара на маркетплейсе  

### Цель фичи:  
Позволить продавцу самостоятельно опубликовать свой товар на маркетплейсе через Личный кабинет продавца.  

### **User story**  
Как продавец маркетплейса, я хочу иметь возможность публиковать товары через личный кабинет, чтобы покупатели могли видеть мои товары на витрине маркетплейса и покупать их.  

### **Use Case**  
Публикация товара на маркетплейсе  

### Актор:   
Продавец  

### Предусловия:  
- Продавец авторизован в личном кабинете
- Продавец имеет доступ к разделу управления товарами  

### Основной сценарий:  
1) Продавец заходит в личный кабинет
2) Открывает раздел "Мои товары"
3) Нажимает кнопку "Добавить товар"
4) Заполняет карточку товара: Название, описание, цену, фото/видео, категории, количество, склад, пометка "Товар 18+"
5) Нажимает кнопку "Опубликовать"
6) Система проверяет данные и регистрирует товар
7) Если нет ошибок - товар становится "Опубликован"
8) Товар отображается на витрине и в поиске

### Постусловия:  
- Товар создан
- Товар получает статус "Опубликован"
- Товар отображается на витрине маркетплейса
- Товар доступен для поиска покупателями  

### Альтернативный сценарий:  
**А1 Сохранение товара как "черновик"**  
  А1.1) Продавец заполняет часть полей  
  А1.2) Продавец нажимает "Сохранить как черновик"  
  А1.3) Система сохраняет введённые данные и товар со статусом "Черновик"   
  А1.4) Товар отображается в списке "Черновики"  
**А1 Постусловия:**  
- Товар создан со статусом "Черновик"  
- Не отображается на витрине  
- Доступен для редактирования

**А2 Автосохранение**  
А2.1) Продавец заполняет форму  
А2.2) Прошло 30 сек без активности  
А2.3) Система автоматически сохраняет черновик  
А2.4) Система показывает индикатор "Автосохранено"  

**A3 Публикация через модерацию**  
А3.1) Продавец нажимает "Опубликовать"  
А3.2) Товар попадает в категорию, требующую проверки  
А3.3) Система присваивает статус "На модерации"  
А3.4) Система показывает сообщение: "Товар отправлен на проверку. Срок: до 24 часов"  
А3.5) Модератор проверяет  
А3.6) Модератор меняет статус на "Опубликован" или "Отклонен"  
А3.7) Система уведомляет продавца  

### Негативный сценарй:  
**Н1 Ошибка валидации обязательных полей**    
Н1.1) Продавец нажимает "Опубликовать"  
Н1.2) Система проверяет обязательные поля  
Н1.3) Обнаруживает ошибки  
Н1.4) Показывает, где данные введены неверно или забыты  
Н1.5) Блокирует публикацию до исправления  

**Н2 Потеря соединения с интернетом**   
Н2.1) Продавец начинает заполнять данные о товаре  
Н2.2) Пропадает интернет  
Н2.3) Система обнаруживает потерю соединения  
Н2.4) Система автосохраняет как черновик последние изменения в создаваемом товаре  
Н2.5) Соединение восстанавливается  
Н2.6) Система показывает: "Соединение восстановлено. Откройте Черновики"  
Н2.7) Продавец нажимает на кнопку "Черновики"  
Н2.8) Выбирает последнее сохранение  
Н2.9) Продолжает заполнять поля товара  

**Н3 Превышен лимит товаров**  
Н3.1) Продавец нажимает "Опубликовать"   
Н3.2) Система проверяет лимиты тарифа   
Н3.3) Если лимит достигнут - блокирует публикацию  
Н3.4) Система выводит сообщение: "Превышен лимит товаров на вашем тарифе. Текущий тариф: (100 товаров). Выберите другой тариф или удалите товары из списка опубликованных"  

### Конечное состояние:  
1) Товар успешно опубликован
   - Товар получает статус "Опубликован"
   - Товар отображается на витрине маркетплейса
   - Товар доступен для поиска покупателями
2) Товар сохранён как черновик
   - Товар имеет статус "Черновик"
   - Товар виден только продавцу в личном кабинете
   - Товар можно редактировать и опубликовать
4) Публикация не выполнена
   - Товар остаётся в состоянии редактирования
   - Продавцу показано сообщение об ошибке
   - Пользователь может исправить данные и повторить публикацию  

### Функциональные требования  
Система должна позволять продавцу:   
1) Создавать карточку товара  
2) Заполнять обязательные поля  
3) Редактировать карточку товара  
4) Публиковать товар на витрине  
5) Сохранять товар в черновик  
6) Удалять товар  
Система должна:  
8) проверять заполненность обязательных полей перед публикацией  
9) проверять товар на соответствие законодательству РФ (Бизнес-ограничения) перед публикацией
10) присваивать маркировку «18+» при публикации товаров соответствующей категории
11) присваивать статусы товару: "Черновик", "На модерации", "Опубликован", "Отклонен"
12) уведомлять продавца о результате публикации (успех/ошибка/модерация)  

### Нефункциональные требования   
1) Время публикации товара системой - не более 5 секунд  
2) Время отклика интерфейса при загрузке формы создания товара - не более 2 секунд 
3) Доступность сервиса публикации товаров - 99,9 % времени
4) Доступ к созданию/редактированию товаров только у авторизованных продавцов с активной ролью
5) Все действия с товаром логируются (кто, когда, что изменил)
6) Поддержка одновременной публикации товаров - до 1000 публикаций в минуту
7) Валидация полей происходит в реальном времени
8) При потере соединения данные формы сохраняются: Черновик сохраняется каждые 30 секунд 
9) Поддержка браузеров: Chrome, Firefox, Safari, Edge (последние 2 версии)

### Бизнес-ограничения  
1) Запрещена публикация товаров, запрещённых к свободной реализации в РФ:
"Федеральный закон от 31 июля 2025 года №289-ФЗ «Об отдельных вопросах регулирования платформенной экономики в Российской Федерации»"  
"Указ Президента РФ от 22 февраля 1992 года №179 «О видах продукции (работ, услуг) и отходов производства, свободная реализация которых запрещена»"

## Задача 3.1  
### Основная информация
| HTTP Метод | POST |
|---------------|---------|
| URL | /api/v1/auth/register (Требует согласования с Backend-lead, возможно "/api/users") |
| Content-Type | application/json |
| Авторизация | Не требуется (публичный эндпоинт) |
### Входные параметры (обязательные поля)
| Название поля | Тип | Описание | Обязательный |
|---------------|---------|----------|------------|
| firstName | String | Имя пользователя | да |
| lastName | String | Фамилия пользователя | да |
| username | String | Логин пользователя | да |
| password | String | Пароль | да |
| captchaToken | String | подтверждение "I am not a robot" (Валидный токен reCaptcha) | да |

Правила валидации пароля:  
- Минимум 8 символов
- Хотя бы 1 цифра (0-9)
- Хотя бы 1 заглавная буква (A-Z)
- Хотя бы 1 строчная буква (a-z)
- Хотя бы 1 специальный символ

**Ограничения остальных полей требуют уточнения у команды.**  
**Но, если взять стандартные параметры ограничений:**  
firstName: 1-50 символов, букв латинского алфавита(A-Z,a-z)    
lastName: 1-50 символов, букв латинского алфавита(A-Z,a-z)    
username: 3-30 символа, букв латинского алфавита(A-Z,a-z) и цифры(0-9), уникальный в системе    
captchaToken: допустимый токен reCaptcha имеет длину около 300–400 символов, но встречаются длиной 2361 символов, должен быть валидирован через Google reCAPTCHA    
### Выходные параметры при успешном ответе (HTTP 200/201)
Предлагаемая структура:  
| Название поля | Тип | Описание | Обязательный | Ограничения |
|---------------|---------|----------|-----------|-----------|
| success | boolean | Флаг | да | Всегда True |
| message | string | Сообщение об успехе | да | текст из предопределённого списка |
| data.userId | string (UUID) | ID созданного пользователя | да | Стандарт UUID v4 |
| data.username | string | Логин пользователя | да | Значение = входному username |
| data.createdAt | string (ISO 8601) | Дата создания аккаунта | да | Формат: YYYY-MM-DDTHH:mm:ssZ (UTC) |
| data.token | string (JWT) | Токен авторизации (если предусмотрен автоматический логин) | нет | Формат: header.payload.signature (base64url) |
### Выходные параметры при ответе с ошибкой (HTTP 4xx/5xx)  
Предлагаемая структура:  
| Название поля | Тип | Описание | Обязательность | Ограничения |
|---------------|---------|----------|----|---------|
| success | boolean | Флаг | да | Всегда False |
| errorCode | string | Код ошибки для разработчиков | да | Значение из фиксированного списка |
| message | string | Сообщение для пользователя | да | Длина: <= 200 символов; без чувствительных данных (паролей, токенов) |
| details | array | Детали валидации (какие поля не прошли) | нет | Макс. 10 элементов |
| timestamp | string (ISO 8601) | Время возникновения ошибки | да | Формат: YYYY-MM-DDTHH:mm:ssZ (UTC) |

| 409 | USER_EXISTS | Клиентская | User exists! |
| 400 | CAPTCHA_INVALID | Клиентская | reCAPTCHA verification failed |
| 400 | CAPTCHA_MISSING | Клиентская | Please verify reCaptcha to register! |
| 422 | USERNAME_INVALID | Клиентская | The username contains invalid characters |
| 500 | INTERNAL_ERROR | Серверная | Internal server error |
| 503 | SERVICE_UNAVAILABLE | Серверная | The service is temporarily unavailable |

## Задача 3.2  
### Алгоритм создания пользователя на бэкенде  
#### Этап 1: Получение и базовая валидация запроса
| Шаг | Действие | Проверка | Результат при ошибке |
|-----|----------|----------|------------|
| 1 | Получить HTTP POST запрос | Метод = POST, URL = /api/v1/auth/register | 405 Method Not Allowed |
| 2 | Проверить заголовки | Content-Type: application/json | 415 Unsupported Media Type |
| 3 | Проверить тело запроса на JSON | Валидный JSON формат | 400 Bad Request + INVALID_JSON |
| 4 | Проверить наличие всех обязательных полей | firstName, lastName, username, password, captchaToken | 400 + MISSING_FIELD |
| 5 | Проверить типы данных полей | Все поля - строки (string) | 400 + INVALID_TYPE |
#### Этап 2: Валидация значений полей  
| Шаг | Действие | Проверка | Результат при ошибке |
|-----|----------|----------|------------|
| 1 | Валидация firstName | Длина 1-50 символов, разрешённые символы | 400 + FIRST_NAME_INVALID |
| 2 | Валидация lastName | Длина 1-50 символов, разрешённые символы | 400 + LAST_NAME_INVALID |
| 3 | Валидация username | Длина 3-30 символов, разрешённые символы | 400 + USERNAME_INVALID |
| 4 | Валидация password | Мин. 8 символов, 1 заглавная, 1 строчная, 1 цифра, 1 спецсимвол | 400 + PASSWORD_WEAK |
| 5 | Валидация captchaToken | Не пустая строка, длина > 0 | 400 + CAPTCHA_MISSING |
#### Этап 3: Проверка безопасности
| Шаг | Действие | Проверка | Результат при ошибке |
|-----|----------|----------|------------|
| 1 | Rate limiting | Не более 5 запросов регистрации в минуту с одного IP | 429 Too Many Requests + RATE_LIMIT_EXCEEDED |
| 2 | Верификация reCaptcha | Отправить токен в Google API, получить success: true | 400 + CAPTCHA_FAILED |
| 3 | Проверка на ботов | Анализ user-agent, поведения, fingerprint | 403 Forbidden + BOT_DETECTED |
#### Этап 4: Бизнес-валидация
| Шаг | Действие | Проверка | Результат при ошибке |
|-----|----------|----------|------------|
| 1 | Проверка уникальности username | Запрос в БД: Существует ли пользователь с указаным в username именем | 409 Conflict + USER_EXISTS |
#### Этап 5: Подготовка данных для сохранения
| Шаг | Действие | Описание |
|-----|----------|----------|
| 1 | Нормализация данных | trim() пробелов, toLowerCase() для username |
| 2 | Хэширование пароля | Использовать bcrypt/argon2 с salt (cost factor 12+) |
| 3 | Генерация UUID | Создать уникальный user_id |
| 4 | Установка метаданных | createdAt, updatedAt, status = 'ACTIVE', emailVerified = false |
#### Этап 6: Сохранение в базу данных
| Шаг | Действие | Проверка | Результат при ошибке |
|-----|----------|----------|------------|
| 1 | Начать транзакцию | BEGIN TRANSACTION | - |
| 2 | INSERT в таблицу users | Вставка записи со всеми полями | ROLLBACK |
| 3 | Записать в audit_log | Логирование действия (кто, когда, IP) | ROLLBACK |
| 4 | Закоммитить транзакцию | COMMIT TRANSACTION | - |
| 5 | Обработка ошибок БД | Unique constraint, connection lost | 500 + DATABASE_ERROR |
#### Этап 7: Пост-обработка
| Шаг | Действие | Описание |
|-----|----------|----------|
| 1 | Отправить приветственное письмо | Асинхронно (очередь задач) |
| 2 | Создать сессию/токен | Если предусмотрен авто-логин (JWT) |
| 3 | Очистить кэш | Инвалидация кэша пользователей |
| 4 | Отправить метрики | user_registered событие |
#### Этап 8: Формирование ответа
| Сценарий | HTTP Status | Тело ответа |
|-----|----------|----------|
| Успех | 201 Created | {"success": true, "message": "...", "data": {...}} |
| Ошибка валидации | 400 Bad Request | {"success": false, "errorCode": "...", "message": "..."} |
| Пользователь существует | 409 Conflict | {"success": false, "errorCode": "USER_EXISTS", ...} |
| Ошибка сервера | 500 Internal Server Error | {"success": false, "errorCode": "INTERNAL_ERROR", ...} |
